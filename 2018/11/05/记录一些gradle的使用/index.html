​ 这时突然又想起了某大佬教我的另一个gradle命令。

于是我在Terminal输入：gradlew app:dep

​ 这个命令能打印出项目的依赖树，能看到所有的依赖，我们找到utils这个库，不出所料，确实是有别的库引用了高版本的utils，导致了utils库升级到了1.0.13。

​

问题找到了，那么应该怎么解决呢？摆在眼前的路有两条。

  1. 修改依赖1.0.13的库到1.0.6保持和线上的版本一致，然后重新上传到maven，然后本地再更新一次，这个方法有点麻烦，但是也是最直接的解决方案。
  2. utils库修复闪退的bug，然后重传到maven，本地更新依赖。这个办法风险很大，因为utils这个库已经加了很多代码了，还要重新拉分支，屏蔽新加的这部分代码。

Gradle命令：

ok，那么上文中所说的gradlew到底是什么呢？gradlew 即gradle wrapper，意思是gradle的一层包装。可以理解成对gradle的封装。你可以在你的项目中找到它，在根目录下有个/gradle/wrapper文件夹，里面有wrapper的jar包和版本号等信息。

常用的gradle命令

  • gradlew -v 可以查看当前gradle的版本号

  • gradlew clean 删除app目录下所有bulid文件夹

  • gradlew build 打包apk，你可以在app/build/outputs/apk下找到输出的包

  • gradlew tasks 列出所有的task

  • build 命令会打出所有buildTypes包，如果你想打包某个buildTypes。你可以使用assemble+buildTypes形式。

    比如：gradlew assembleDebug gradlew assembleRelease等。

    此命令更厉害的地方在于还能打渠道包，我们以wandoujia渠道为例。你可以使用assemble+Product Flavor+buildTypes

    比如：gradlew assembleWandoujiaRelease。直接就能输出豌豆荚渠道下的realease包。

  • graldew installRelease 打包并安装release包

  • gradlew uninstallRelease 卸载Release包

  • gradlew app:dep 查看项目依赖树

排除传递依赖

  • exclude：排除指定模块的依赖。参数分别是group和module，会排除所有匹配项

  • transitive：默认为true，gradle自动添加子依赖项。设置为false排除所有的传递依赖，可以用来解决一些依赖冲突的问题。

  • force：强制设置某个模块的版本。
